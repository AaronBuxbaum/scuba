# 0005 — Use Postgres via Drizzle ORM, with PGlite for dev and tests

- **Status:** Accepted
- **Date:** 2026-07-17

## Context

M1 needs the domain spine (shops, people, trips, bookings) persisted. Agents develop in
ephemeral sandboxes where running a database daemon is friction and often impossible; yet the
product is a multi-tenant SaaS that will want real Postgres semantics (enums, timestamptz,
transactional integrity) in production.

## Decision

- **Postgres dialect** everywhere, defined and queried through **Drizzle ORM**
  (`src/db/schema.ts` is the single source of truth; SQL migrations generated by `drizzle-kit`
  into `drizzle/` and committed).
- **PGlite** (embedded WASM Postgres, `@electric-sql/pglite`) as the driver for local dev and
  tests: zero daemons, zero setup — `pnpm dev` and `pnpm test` work in any sandbox. Data lives
  in `.pglite/` (gitignored); tests use in-memory instances.
- In dev/test the client **auto-migrates and auto-seeds** on first connection, so a fresh
  checkout boots into a working demo shop with no manual steps.
- Production will swap the driver to a server Postgres (`drizzle-orm/node-postgres`) behind the
  same `getDb()` seam when hosting lands (deferred decision).

## Alternatives considered

- **Prisma** — great DX but engine binaries and generate steps are friction in sandboxes;
  schema DSL is a second language where Drizzle is plain TS.
- **SQLite (better-sqlite3/libsql)** — simplest dev story, but diverges from production
  Postgres semantics exactly where it hurts (enums, timezones, concurrent writes).
- **Dockerized Postgres for dev** — not available in agent sandboxes; violates the
  zero-setup requirement.

## Consequences

- Agents get a real Postgres-flavored database with no environment setup; CI needs nothing.
- PGlite is single-connection and not for production traffic — acceptable: it is a dev/test
  vehicle only. The production driver swap is isolated to `src/db/client.ts`.
- Schema changes follow: edit `schema.ts` → `pnpm db:generate` → commit the SQL migration with
  the change. Never hand-edit applied migrations.
