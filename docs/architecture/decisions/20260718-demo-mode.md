# 20260718-demo-mode — Ship a gated, resettable demo shop for self-serve trials

- **Status:** Accepted
- **Date:** 2026-07-18

## Context

A prospective dive-shop owner wants to feel the staff surfaces before committing — not read a
feature list, not book a sales call. We already seed one example shop (Blue Mantis Divers) for dev
and e2e, and the staff surfaces are real. What's missing is a way to let a stranger *into* those
surfaces safely. The constraints: no signup flow yet (M7 territory), a single-shop instance for now
(multi-shop routing is deferred, `docs/product/roadmap.md`), and an ironclad guarantee that the
demo's privileged entry can never appear in a real customer's production instance.

## Decision

- **Demo mode is a deployment-level flag**, `isDemoMode()` in `src/lib/demo.ts`: on everywhere
  except production by default, forced either way by `SCUBA_DEMO` (`1/true/on` vs `0/false/off`).
  A dedicated demo deployment sets `SCUBA_DEMO=1` in production; a real shop leaves it unset.
- **The seeded Blue Mantis shop *is* the demo shop** (`DEMO_SHOP_SLUG`). No new tenant, no
  per-visitor isolation — everyone shares one example shop.
- **One-click entry signs in as the seeded owner** via the ordinary Credentials flow
  (`enterDemoAction`). It surfaces a known login; it does not bypass auth (ADR-0006). The CTA and
  action are both gated by `isDemoMode()`.
- **Reset restores the playground, not the shop.** `seed.ts` splits into a stable half (shop,
  staff, logins) and a resettable half (`seedDemoSchedule`: customers, trips, bookings).
  `resetDemoSchedule` wipes the resettable half — including walk-ups the booking flow created —
  and re-seeds it, leaving staff and their logins untouched so the demo session survives its own
  reset. `resetDemoAction` is staff-gated *and* demo-gated.
- **The staff surfaces wear the demo badge**: `/shop/layout.tsx` renders a `DemoBanner` (with the
  reset button) whenever demo mode is on.

## Alternatives considered

- **Per-visitor ephemeral shops** — the "real" isolated sandbox, but needs multi-tenant routing
  and per-session provisioning that the roadmap explicitly defers. Revisit when shop onboarding
  lands and multi-shop routing exists.
- **Passwordless magic entry (skip Credentials)** — fewer moving parts, but forks the auth path
  and invites a bypass that could leak past the gate. Reusing the seeded login keeps one auth road.
- **Read-only demo** — safe but lifeless; the whole point is letting someone schedule a trip and
  cancel a booking. Reset makes a writable shared shop tolerable.

## Consequences

- A stranger can drive the real staff UI in one click, and tidy up after themselves.
- The shared shop means one visitor's edits are visible to the next until someone resets. Accepted
  for a trial; the escape hatch is per-visitor shops once multi-tenancy exists.
- The demo owner login is a real seeded credential. It only matters where `isDemoMode()` is true;
  a production shop that never sets `SCUBA_DEMO` never exposes the entry. If we ever want the demo
  in production without the dev seed running, we would provision the demo shop explicitly rather
  than rely on `seedIfEmpty`.
- No new runtime dependency; the change is a gate, a seed split, and three thin server actions.
